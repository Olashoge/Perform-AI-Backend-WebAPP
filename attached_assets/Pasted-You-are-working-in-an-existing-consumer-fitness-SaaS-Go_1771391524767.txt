DATA MODEL (PRISMA)
Implement a robust model for workout exercise feedback.

A) If the app already has a “Preference” or “UserFeedback” table, extend it carefully. Otherwise add:

model ExercisePreference {
  id            String   @id @default(cuid())
  userId        String
  exerciseKey   String   // stable key/slug e.g. "barbell_back_squat" or normalized name hash
  exerciseName  String   // display name, e.g. "Back Squat"
  status        ExercisePreferenceStatus // LIKED | DISLIKED | AVOIDED
  createdAt     DateTime @default(now())
  updatedAt     DateTime @updatedAt

  @@unique([userId, exerciseKey])
  @@index([userId, status])
}

enum ExercisePreferenceStatus {
  LIKED
  DISLIKED
  AVOIDED
}

B) If exercises already have IDs in your DB, prefer referencing exerciseId instead of exerciseKey.
But if exercises are generated dynamically, use exerciseKey derived from name normalization:
- lowercase
- trim
- replace spaces/punctuation with underscores
- collapse repeated underscores

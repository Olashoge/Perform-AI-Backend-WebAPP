D) TRUE AUTOMATIC WEEKLY PLAN ADAPTATION LOGIC (V1)
========================================================

Objective:
When a week ends (or on weekly check-in), automatically adapt the NEXT week’s plan generation parameters based on adherence + friction + profile flags.

Define signals (inputs):
- mealCompletionRate (0–1) from checkboxes
- workoutCompletionRate (0–1)
- optional energy rating (if you have it)
- friction indicators:
  - repeated skips on same meal type (breakfast/lunch/dinner)
  - repeated skipped workouts on same day
  - swaps/regens usage (proxy for dissatisfaction)
- profile flags:
  - appetiteLevel
  - bodyContextNotes keywords (hardgainer, busy, low appetite)
  - goal type (muscle gain vs fat loss vs performance)

Adaptation rules (simple + explainable):
1) If mealCompletionRate < 0.60:
  - Reduce complexity:
    - fewer unique recipes, more repeats
    - shorter prep time emphasis
  - Increase calorie density if goal is muscle gain OR low appetite flagged
  - Add 1 “favorite meal” daily in healthier version
2) If workoutCompletionRate < 0.60:
  - Reduce volume slightly:
    - fewer exercises per session OR fewer sets
  - Increase equipment alignment:
    - prefer machines if gym selected; prefer dumbbells/bands for home
  - Keep intensity moderate; prioritize consistency
3) If both meal and workout > 0.80:
  - Progressive overload / progression:
    - workouts: slight volume or intensity increase
    - nutrition: small adjustment toward goal (calorie surplus/deficit tuning)
4) If energy low (if available):
  - Reduce intensity and increase recovery emphasis next week.

Where adaptation is stored:
- Create table: weekly_adaptations
  - userId
  - weekStart
  - computedSignals JSON
  - adaptationParams JSON (what changed: complexity, volume, calorie density, favorite meal frequency, etc.)
  - createdAt
- Next plan generation uses latest adaptationParams as additional constraints.

UX:
- On Weekly Check-in / Progress:
  - Show a small “Next week adjustments” summary (plain language):
    - “Simplifying meals for easier consistency”
    - “Slightly reducing workout volume to improve completion”
    - “Including favorite meals more often”
No technical jargon.

Trigger:
- When user submits weekly check-in OR when week rolls over and we have enough completion data, generate/store adaptationParams.

Safety:
- Never auto-change user profile.
- Only affect next week generation defaults and prompts.
- Users can always regenerate.

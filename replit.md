# Perform AI

## Overview
Perform AI is an AI-powered personal performance system that generates personalized 7-day meal and workout plans. Users input their dietary and fitness preferences, along with personalization data such as age, weight, and workout frequency. The system then leverages AI to create customized 7-day meal plans, complete with step-by-step recipes, nutrition information, and organized grocery lists. Additionally, it generates tailored workout programs. The project aims to provide a comprehensive solution for individuals seeking to optimize their personal health and fitness routines through intelligent, adaptive planning.

## System Architecture
The application is built with a modern web stack, emphasizing a responsive and intuitive user experience. The frontend is developed using React with Vite and TypeScript, styled with Tailwind CSS and shadcn/ui components. The UI/UX features a premium visual redesign, including an icon-only navigation sidebar and an active goal bar.

The backend is an Express server written in TypeScript, managing session-based authentication with PostgreSQL for session storage, ensuring 30-day session persistence. PostgreSQL is also used as the primary database, interfaced via Drizzle ORM.

Core features include:
- **Personalized Plan Generation**: AI-generated 7-day meal and workout plans based on user preferences and personal data. Meal plans are configurable for 2 or 3 meals per day, with dynamic meal slot selection. Workout plans consider goals, location, training modes, focus areas, session length, experience level, and injuries.
- **Goal-Oriented Planning**: A `GoalPlan` entity orchestrates the generation of both meal and workout plans, supporting various goal types (e.g., weight loss, muscle gain). It features a sequential generation pipeline (workout → meal plan → scheduling → finalizing) with real-time progress tracking.
- **Feedback Loop & Adaption**: Users can provide tri-state feedback (like/dislike/neutral) on individual meals and workout sessions. This feedback influences future AI-generated plans by tracking ingredient and exercise preferences/avoids.
- **Grocery Management**: AI-estimated grocery pricing with confidence levels, and a feature for users to track owned items, dynamically updating totals. Grocery lists can be rebuilt from current meals.
- **Scheduling & Calendar**: Plans are schedulable with conflict-aware date pickers. A unified calendar view displays all scheduled meal and workout plans.
- **Wellness Context**: A shared `WellnessContext` on the server ensures cross-plan coupling, adapting nutrition based on workout schedules (e.g., increased carbs/calories on training days).
- **Check-ins**: Weekly check-in logging for weight, energy, compliance, and notes.
- **Adaptive Swap/Regen Economy**: Dynamic allowance system anchored to GoalPlans. Daily-resetting counters for meal swaps (default 2/day) and day regens (default 1/day), with plan-wide regen budgets (default 5). Enforces 6-hour cooldowns after 3 regens in 24 hours. Adaptive bonuses/penalties computed from behavior summaries (adherence, feedback patterns) carry forward to the next plan. Flex tokens earned for 7+ day streaks can be redeemed for extra regens. UI panel ("Today's Budget") on plan-view.tsx shows live counters, cooldown status, and coach insights. Workout session regeneration supported via POST `/api/workout/:id/regenerate-session` with Zap icon button on each SessionCard. Workout regens share the unified daily regen counter and plan-wide budget with meal regens. Tables: `plan_allowances`, `plan_usage_events`, `flex_tokens`, `plan_behavior_summaries`. Service: `server/allowance.ts`.
- **Profile (Performance Blueprint) — Single Source of Truth**: Mandatory user profile (`user_profiles` table) capturing physical stats (age, sex, height, weight, target weight), primary goal, training experience, health conditions (`healthConstraints` jsonb — unified from injuries, mobility limitations, chronic conditions; sleep, stress), training capacity (activity level, `trainingDaysOfWeek` as jsonb day array e.g. ["mon","wed","fri"], session duration), nutrition preferences (`allergiesIntolerances` jsonb — unified from allergies + intolerances; appetite, spice preference), and dietary restrictions (`foodsToAvoid` jsonb array + `foodsToAvoidNotes` text). Legacy columns (`allergies`, `intolerances`, `injuries`, `mobilityLimitations`, `chronicConditions`) are still populated on save for backward compatibility but canonical fields are `allergiesIntolerances` and `healthConstraints`. `religiousRestrictions` UI removed (column retained). Profile is **required before any plan creation** — enforced on all routes: `/api/goal-plans/generate`, `/api/plan`, `/api/workout` (server returns 400 with `profileRequired: true`). **Profile Prefill**: All plan creation forms (new-plan, new-workout, goal-wizard) auto-prefill from profile data using `useEffect` + `useRef` guard pattern (goal mapping, allergiesIntolerances, foodsToAvoid, spiceLevel, trainingDays, sessionLength, experienceLevel, healthConstraints). Standardized "Your Profile" summary cards display across all 3 pages with consistent heading, layout, and enriched fields (allergies & intolerances, foods to avoid, health constraints). `householdSize` hidden behind "Show advanced options" toggle (defaults to 1). Pace field includes descriptive helper text. A `profileSnapshot` (jsonb) is stored with each `GoalPlan`, `MealPlan`, and `WorkoutPlan` for historical tracking. Profile page uses interactive day-chip toggles for `trainingDaysOfWeek` and chip selection for `foodsToAvoid`. Page: `client/src/pages/profile.tsx`. Routes: `GET/POST/PUT /api/profile`.
- **Constraint Engine v1**: Deterministic safety and rules layer that runs before AI generation (pre-checks) and validates after (post-checks). Pre-checks evaluate user profile against safety rules (age restrictions, injury/mobility bans, equipment availability, nutrition bounds, schedule overlap) and produce a `SafeSpec` that constrains AI prompts. Post-validation scans AI output for banned ingredients/exercises and auto-fixes small violations or triggers regeneration. BLOCK severity violations return 400 to the client with violation details. Tables: `constraint_violations`, `wellness_plan_specs`. Modules: `server/constraints/types.ts`, `server/constraints/engine.ts`, `server/constraints/post-validation.ts`, `server/constraints/rules/` (age, injury, equipment, nutrition, schedule). Integrated into `/api/goal-plans/generate`, `/api/plan`, `/api/workout` routes. UI displays blocked violations via toast messages in goal wizard, new-plan, and new-workout pages.
- **Performance Core v1**: Weekly performance summary system triggered by check-ins. Computes adherence scores (weighted: 40% meals, 40% workouts, 20% energy), momentum states (building/maintaining/fatigue_risk/slipping), insights, and adjustment actions (maintain/reduce_load/increase_load/simplify_plan/nutrition_bias_training_days). Economy deltas (+1 regen/swap bonus for 2+ weeks at 85+, -1 penalty for 2+ weeks below 60) carry forward to next plan's allowance. The `nextWeekPlanBias` field on `user_profiles` stores the latest adjustment action and is injected into AI prompts via the Constraint Engine's `buildConstraintPromptBlock`. Tables: `performance_summaries`. Module: `server/performance/computeWeeklySummary.ts`. Routes: `GET /api/performance/latest`, `GET /api/performance`. UI: PerformanceSummaryCard on check-ins page.
- **Daily Planning ("Plan this day")**: Single-day meal and workout planning via bottom sheet UI. Users click "Plan this day" from Dashboard week strip (empty or populated days) to open a bottom sheet with options to generate a daily meal plan (2 or 3 meals) or a daily workout. Generation is async with polling status. Tables: `daily_meals` (unique per user+date, stores planJson with meals and groceryJson), `daily_workouts` (unique per user+date, stores planJson with workout session). AI functions: `generateSingleDayMeals`, `generateSingleDayWorkout` in `server/openai.ts`. Routes: `POST /api/daily-meal`, `POST /api/daily-workout`, `GET /api/daily-meal/:date`, `GET /api/daily-workout/:date`, `GET /api/daily-meals?start&end`, `GET /api/daily-workouts?start&end`, `GET /api/daily-coverage?start&end`. Frontend: `PlanThisDay` sheet component (`client/src/components/plan-this-day.tsx`), `DailyMealView` page (`client/src/pages/daily-meal-view.tsx`), `DailyWorkoutView` page (`client/src/pages/daily-workout-view.tsx`). Integrated into Dashboard with coverage-aware indicators. Profile required for generation (enforced server-side).
- **Adaptive Engine v1**: Deterministic rule-based system that automatically adjusts plan difficulty, volume, and complexity based on user performance history. Computes `AdaptiveModifiers` (volumeMultiplier, intensityCapRPE, cardioBias, recoveryBias, complexityLevel, nutritionCalorieDeltaKcal, trainingDayCarbBias, simplifyMeals, deloadWeek) from recent performance summaries, check-ins, profile, and pace. Decision codes (e.g., LOW_WORKOUT_ADHERENCE_SIMPLIFY, FATIGUE_RISK_DELOAD, HIGH_WORKOUT_ADHERENCE_PROGRESS) map to user-friendly messages for UI transparency. An `adaptiveSnapshot` (jsonb) is persisted on all plan tables (mealPlans, workoutPlans, goalPlans, dailyMeals, dailyWorkouts) alongside profileSnapshot. Modifiers are injected into AI generation prompts via `buildAdaptivePromptBlock`. Modules: `server/adaptive/types.ts`, `server/adaptive/computeAdaptiveModifiers.ts`, `server/adaptive/buildAdaptivePromptBlock.ts`, `server/adaptive/decisionMessages.ts`, `server/adaptive/index.ts`. UI: `AdaptiveInsightsCard` component (`client/src/components/adaptive-insights-card.tsx`) displayed on plan-view, workout-view, daily-meal-view, and daily-workout-view pages with collapsible "Why this plan looks like this" section showing decision messages and modifier badges. Progression/Deload badges on workout plan headers. Recent Progress section on check-ins page showing last 4 performance summaries with momentum states.
- **Rate Limiting**: AI calls are rate-limited to 10 per user per day to manage resource usage.
- **Soft Deletion**: Plans and goals are soft-deleted, maintaining data integrity while allowing for recovery.

## External Dependencies
- **OpenAI**: Utilizes `gpt-4.1-mini` for generating personalized meal plans and workout programs.
- **PostgreSQL**: Primary database for all application data, including user information, session storage, plans, preferences, and check-ins.
- **connect-pg-simple**: Used for PostgreSQL-backed session storage in the Express backend.
